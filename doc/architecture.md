
# Архитектура и требования

Этот документ является единым источником для:

- требований (функциональных/нефункциональных),
- архитектуры и ключевых решений,
- диаграмм (с описаниями),
- обоснования выбора технологий,
- декомпозиции задач между участниками команды.

## 1. Требования

### 1.1 Область задачи

NetChat — консольный peer-to-peer чат с прямыми TCP-соединениями между узлами (без центрального сервера). Каждый экземпляр приложения является узлом сети и может:

- принимать входящие подключения (режим ожидания),
- подключаться к указанному peer-у (режим подключения),
- отправлять и принимать текстовые сообщения.

### 1.2 Термины

- **Узел (node)** — запущенный экземпляр NetChat.
- **Peer** — удалённый узел, к которому мы подключаемся или который подключился к нам.
- **Сообщение чата** — пользовательское сообщение (текст), которое отображается с именем отправителя и датой/временем.

### 1.3 Функциональные требования

**F1. Параметры запуска**

При запуске пользователь задаёт:

- имя пользователя (обязательно),
- адрес peer-а (опционально) — если указан, узел пытается подключиться,
- адрес прослушивания (опционально).

Минимальный набор аргументов:

- `--name <username>`
- `--peer <host:port>` (необязательно)
- `--listen <host:port>` (необязательно)

Если `--peer` не задан, узел работает в режиме ожидания входящих подключений.

**F2. Отправка сообщений**

Пользователь вводит текст в консоли; каждое отправленное сообщение доставляется подключённым peer-ам.

**F3. Приём и отображение сообщений**

Каждое полученное сообщение отображается в консоли со следующими полями:

- имя отправителя,
- дата/время отправки,
- текст сообщения.

**F4. Формат времени**

Рекомендуемый формат отображения: `yyyy-MM-dd HH:mm:ss` (локальное время). В протоколе время передаётся как ISO-8601 timestamp.

**F5. Корректное завершение работы**

Узел корректно закрывает соединения при завершении.

**F6. Обработка ошибок**

Должны быть обработаны и понятным образом показаны пользователю:

- неверный формат аргументов,
- невозможность забиндить порт,
- невозможность подключиться к peer-у,
- разрыв соединения,
- получение некорректного сообщения.

### 1.4 Нефункциональные требования

**N1. Портируемость:** Windows/Linux/macOS при наличии Java 21.

**N2. Ограничения:**

- базовый сценарий: 2 узла и 1 соединение;
- длина текста: до 4096 символов (лимит фиксируется в коде);
- входные данные валидируются, приложение не падает от мусора.

**N3. Надёжность:** сетевые ошибки не приводят к аварийному завершению.

**N4. Наблюдаемость:** логируются start/stop, connect/disconnect, ошибки кодека/транспорта.

## 2. Цели архитектуры

- Реализовать консольный peer-to-peer чат с прямым соединением по сети.
- Обеспечить читаемую архитектуру без «архитектурной жадности».
- Сделать проект проверяемым: тесты + CI, предсказуемый стиль кода.

## 3. Выбор технологий и обоснование

### 3.1 Java 21 + Maven

Выбор Java обусловлен:

- знакомостью команды (снижает риски по срокам),
- богатой стандартной библиотекой для сетей и многопоточности,
- зрелой экосистемой тестов и статического анализа.

Maven выбран как простой и распространённый сборщик с хорошей интеграцией в CI.

### 3.2 TCP + NDJSON (база)

Для базовой версии используется TCP и простой текстовый протокол поверх него:

- **NDJSON** (newline-delimited JSON): каждое сообщение — JSON-объект в одной строке, разделитель — `\n`, кодировка — UTF‑8.

Причины:

- легко отлаживать «глазами» и логами,
- не требует дополнительного IDL/генерации,
- достаточно для учебного проекта.

### 3.3 gRPC (опционально)

gRPC рассматривается только как расширение, если останется время. В базовом объёме от него отказываемся, чтобы не усложнять:

- деплой и отладку,
- протокол (framing/HTTP2),
- тестирование и совместимость.

## 4. Диаграммы (с пояснениями)

Диаграммы хранятся в `doc/diagrams/` (PlantUML):

- `use_cases.puml` — пользователь запускает узел, выбирает режим listen/connect, отправляет и получает сообщения.

## 5. Компонентная декомпозиция

Минимальные компоненты (не фреймворк, а слои/модули внутри приложения):

1) **CLI/UI (console)**
	- парсинг аргументов
	- чтение stdin
	- форматированный вывод сообщений и статусов

2) **Chat Core (домен)**
	- модель сообщения
	- маршрутизация входящих/исходящих сообщений
	- управление жизненным циклом соединений

3) **Transport (TCP)**
	- сервер: accept входящих соединений
	- клиент: connect к peer-у
	- чтение/запись потоков

4) **Protocol (codec)**
	- сериализация/десериализация сообщений
	- валидация ограничений (длина, обязательные поля)

В дальнейшем допускается добавление небольших утилитных компонентов:

- `Clock`/`TimeProvider` для тестируемости времени
- `Logger` (в базовом варианте — `java.util.logging`)

## 6. Протокол сообщений

### 6.1 Формат

Каждое сообщение передаётся как одна строка JSON (без переносов внутри):

```json
{"type":"chat","sender":"Alice","sentAt":"2026-03-30T21:15:02.123+03:00","text":"Привет"}
```

Поля:

- `type`: строка, сейчас фиксировано `chat` (задел на служебные сообщения)
- `sender`: имя пользователя
- `sentAt`: timestamp ISO-8601 (OffsetDateTime)
- `text`: текст сообщения

### 6.2 Ограничения

- Максимальная длина `text`: 4096 символов
- Максимальная длина строки на проводе (line length): фиксируется в коде (например, 8–16KB) для защиты от DoS/ошибок

### 6.3 Поведение при ошибках

- Невалидный JSON/поля: сообщение игнорируется, ошибка логируется; соединение может быть закрыто (решение фиксируется при реализации).

## 7. Параллелизм и модель выполнения

Требование: консольный ввод и сетевой ввод/вывод не должны блокировать друг друга.

Рекомендуемая модель:

- **Thread A:** читает stdin, формирует исходящие сообщения и отправляет в Chat Core.
- **Thread(s) B:** на каждое соединение — поток чтения (или общий пул), который читает строки из сокета и передаёт в Chat Core.
- **Thread(s) C:** поток(и) записи могут быть общими через очередь на соединение, чтобы сериализовать writes.

В базовой версии допустима модель «один поток на соединение», чтобы сократить сложность.

## 8. Границы ответственности (классы)

Ориентир для диаграммы классов:

- `App` — точка входа, сборка зависимостей
- `CliArgs`/`ArgsParser` — разбор `--name`, `--peer`, `--listen`
- `ChatNode` — основной фасад: старт/стоп, подключение, обработка событий
- `PeerConnection` — абстракция соединения (read/write, идентификатор)
- `Message` — модель сообщения
- `MessageCodec` — encode/decode + валидация
- `TcpServer` — accept loop
- `TcpClient` — connect

## 9. Декомпозиция задач на 2 участников

### Участник A (Сеть/протокол)

- реализация TCP-транспорта (сервер/клиент)
- кодек протокола (NDJSON), лимиты, обработка ошибок
- интеграционные тесты transport/protocol

### Участник B (CLI/ядро/качество)

- CLI (аргументы, пользовательский ввод/вывод)
- Chat Core (маршрутизация, жизненный цикл)
- документация, styleguide, CI, базовые unit-тесты

Перекрытие допустимо, но желательно держать границы ответственности для PR-ревью.

## 10. Риски и упрощения

- P2P в полном смысле (NAT traversal, discovery) не реализуется — только прямое соединение по адресу.
- Без шифрования/аутентификации в базовой версии.
- Масштабирование на множество peer-ов — опционально; базовый сценарий: 1 соединение.
