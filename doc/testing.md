# План тестирования

## 1. Цели тестирования

- Проверить корректность протокола сообщений и его устойчивость к ошибкам входных данных.
- Проверить корректность CLI (аргументы запуска, режимы listen/connect).
- Проверить базовый сценарий peer-to-peer: отправка/приём сообщений между двумя узлами.
- Обеспечить автоматический прогон основного набора тестов в CI.

## 2. Объекты тестирования

- Парсер аргументов (`--name`, `--peer`, `--listen`) и валидация.
- Кодек протокола (NDJSON): сериализация/десериализация, лимиты.
- Транспорт TCP: подключение, чтение/запись, разрыв соединения.
- Логика ядра (Chat Core): маршрутизация сообщений, жизненный цикл соединений.

## 3. Уровни тестов

### 3.1 Unit-тесты (обязательный костяк)

- `ArgsParser`:
  - имя обязательно;
  - корректный/некорректный `host:port`;
  - поведение при одновременных флагах.
- `MessageCodec`:
  - encode/decode round-trip;
  - невалидный JSON;
  - отсутствующие поля;
  - лимит длины `text`.

### 3.2 Интеграционные тесты (обязательный костяк)

- «два узла на localhost»:
  - узел A слушает порт;
  - узел B подключается;
  - B отправляет сообщение → A отображает;
  - A отправляет ответ → B отображает.

### 3.3 Системные/ручные проверки (план)

- Проверка интерактивного ввода в консоли.
- Проверка читаемости вывода (формат времени, имя отправителя).
- Проверка на Windows и Linux.

## 4. Тестовый чеклист (что должно быть в репозитории)

Отмечайте галочками по мере реализации:

- [ ] Unit: парсинг аргументов
- [ ] Unit: кодек сообщений (валидные сообщения)
- [ ] Unit: кодек сообщений (невалидные сообщения)
- [ ] Unit: лимит длины текста
- [ ] Integration: connect + send/receive на localhost
- [ ] Integration: разрыв соединения не роняет процесс
- [ ] Manual: 2 процесса на одной машине
- [ ] Manual: 2 машины в одной сети

## 5. Запуск тестов

Локально:

- `mvn -B -ntp test`

В CI:

- `mvn -B -ntp verify`
